# 六爻占卜网页 · 设计文档

- **日期**：2026-06-16
- **名称**：六爻
- **一句话**：一个时髦的六爻（周易金钱卦）算命网页，托管在 GitHub Pages，纯前端静态站，留好接口供二期接入 AI 解卦。

---

## 1. 目标与定位

做一个**视觉高级、移动端优先**的六爻起卦小应用：用户写下所问之事 → 摇卦（三铜钱六掷）→ 得本卦（含变卦）→ 看卦象与周易原文 → 一键生成精致分享卡片。

核心取舍：

- **纯静态优先**：六十四卦与周易卦爻辞是公版固定内容，全部打包进站内，零后端、零成本、可离线，完美契合 GitHub Pages。
- **为 AI 预留接缝**：把「解卦」抽象成一个接口；一期返回本地周易原文，二期把该接口换成调 serverless 的 AI 个性化解读，前端其余部分基本不动。

---

## 2. 范围

### 2.1 一期做（MVP）

1. **输入所问之事**——摇卦前让用户写下想问的问题。
2. **摇卦**——三枚铜钱抛掷动画，摇六次，自下而上成卦。
3. **出卦**——展示本卦（及有动爻时的变卦）、卦名、六爻、动爻 / 世应标记，并显示**周易卦辞 + 动爻爻辞原文**（公版固定文字）。
4. **分享卡片**——一键生成一张含「所问之事 + 卦象结果」的精致图片，可保存 / 转发。

### 2.2 一期不做（明确排除）

- 现代白话解读（分事业 / 感情 / 财运等维度）——**正好留给二期由 AI 生成**。
- 摇卦历史 / 收藏、用户账号、每日一卦推送。
- 纳甲、六亲、六神等专业六爻排盘（保持「时髦轻量」而非「专业排盘」）。

### 2.3 二期预留（不在本次实现，仅设计接缝）

- AI 个性化解读：用户的「所问之事」+ 卦象 → serverless（Cloudflare Workers / Vercel Functions）转发 LLM → 返回结构化解读。
- 接缝点：`interpret()` 接口（见 §7.4）。

---

## 3. 用户流程

```
起卦页（输入所问） → 摇卦页（铜钱六掷动画，逐爻成卦） → 出卦页（卦象 + 周易原文） → 分享卡片
```

线性、单向、一次一屏。状态机：`input → casting → result`。

---

## 4. 视觉设计：极简 · 宣纸白

气质基调：**宣纸米白底 + 细墨线卦象 + 朱砂红单焦点 + 衬线 + 大量留白**。安静、克制、文艺，性冷淡的高级感。移动端优先。
（注：构思阶段曾选「墨金·内敛」深色方案，实现并预览后改定为本「极简·宣纸白」方向。）

### Design Tokens（落到 Tailwind theme / CSS 变量）

| 用途 | 值 |
|---|---|
| 背景（宣纸米白） | `radial-gradient(circle at 50% 18%, #faf7ef, #ece5d4)`；卡片 / 分享图 `#f5f1e7` |
| 主墨（卦象线条 / 卦名 / 正文，token `ink`） | `#1b1b1b` |
| 次级文字 | `ink` + 透明度（`text-ink/50~70`） |
| 朱砂红唯一焦点（动爻「动」标 / 世应 / 卦辞标签 / 印，token `seal`） | `#c0392b` |
| 衬线字体（卦名 / 卦辞 / 爻辞） | `'Songti SC','STSong', serif`；二期再打包子集化宋体 webfont 保证分享图跨设备一致 |

风格约束：细墨线、大留白、无辉光、不堆装饰。卦象每行定宽三栏对齐（动爻点｜定宽爻线｜世应），各爻等距；卦名置于各自卦象正上方（本卦 / 变卦分列）。

## 5. 技术选型

- **构建**：Vite
- **框架**：React + TypeScript
- **样式**：Tailwind CSS
- **动画**：Framer Motion（铜钱抛掷、逐爻成卦、转场）
- **分享图**：`modern-screenshot`（对 CJK / webfont 处理优于旧的 html-to-image），把隐藏的 ShareCard DOM 在客户端栅格化为 PNG，零后端
- **部署**：GitHub Actions 自动 build → GitHub Pages；Vite 设 `base: '/<仓库名>/'`
- **随机源**：`crypto.getRandomValues`（公平，且可在测试中注入 mock）

## 6. 架构与目录结构

**核心原则：领域逻辑（起卦 / 查卦 / 解卦）全部为纯函数，与 React 解耦**——便于测试，也便于二期替换解卦实现。多小文件、高内聚低耦合。

```
src/
  domain/
    types.ts            # Line / Hexagram / Reading 等类型
    lines.ts            # 爻的类型与阴阳/动静判定
    casting.ts          # 起卦：随机源 → 六掷 → 本卦 → 动爻 → 变卦
    random.ts           # RandomSource 接口 + crypto 实现（可注入 mock）
    hexagram-lookup.ts  # 六爻二进制 → 卦 的查表
    interpret.ts        # 解卦接缝（一期=本地原文；二期=调 serverless）
  data/
    hexagrams.ts        # 六十四卦数据集（见 §8）
  components/
    QuestionInput.tsx   # 输入所问之事
    CoinToss.tsx        # 单次三铜钱抛掷动画
    CastingStage.tsx    # 摇卦页：编排六掷、逐爻成卦
    Hexagram.tsx        # 纯展示：渲染六爻（阴阳/动爻/世应标记），多处复用
    ResultView.tsx      # 出卦页：本卦/变卦 + 卦名 + 卦辞 + 动爻爻辞
    ShareCard.tsx       # 分享卡片（离屏渲染，供栅格化）
  hooks/
    useCasting.ts       # 串起状态机 input→casting→result
    useShareImage.ts    # 触发 DOM→PNG、保存/分享
  App.tsx
  main.tsx
```

## 7. 领域模型与核心算法

### 7.1 爻（Line）

- 阴 / 阳，少 / 老（老 = 动爻）。
- 表示：用一位二进制表示阴阳（阳=1，阴=0），另用标志位表示「是否动爻」。

### 7.2 起卦规则（写死并测试）

摇一次 = 抛三枚铜钱。**字面记 2，背面记 3**，三枚相加：

| 三枚 | 和 | 名 | 阴阳 | 动静 |
|---|---|---|---|---|
| 字字字 | 6 | 老阴 | 阴爻 | **动**（变阳）|
| 字字背 | 7 | 少阳 | 阳爻 | 静 |
| 字背背 | 8 | 少阴 | 阴爻 | 静 |
| 背背背 | 9 | 老阳 | 阳爻 | **动**（变阴）|

规律：**和的奇偶定阴阳**（奇=阳，偶=阴）；**6 与 9（三枚全同）为动爻**。等价口诀：数「背」的个数，偶数(0/2)=阴、奇数(1/3)=阳；全同才动。

摇六次，**自下而上**（初爻→上爻）叠成**本卦**。

### 7.3 变卦

把本卦中所有动爻（老阴 6 / 老阳 9）翻面（阴↔阳），得到**变卦**。无动爻则无变卦。

### 7.4 查卦与世应

- 把六爻阴阳转成二进制键，查 `hexagrams.ts` 得到：卦名、上下卦、卦辞、六条爻辞、**世爻位 / 应爻位**（八宫固定，作为数据存储）、卦符。
- 世应位置随卦固定，直接存进数据集，避免运行时推算。

### 7.5 解卦接缝 `interpret()`

```ts
// 一期与二期共用同一签名，从一开始就 async，避免二期改签名
interpret(reading: CastReading): Promise<Interpretation>
```

- **一期实现**：同步组装——从数据集取本卦卦辞 + 各动爻爻辞原文，按传统取用规则决定展示哪些爻辞（见下），包成 `Interpretation` 返回。
- **二期实现**：换成 `fetch` 调 serverless，传「所问 + 卦象」，返回 AI 结构化解读。前端调用处不变。
- **取爻辞规则（一期展示用，传统简化版）**：无动爻→看本卦卦辞；有动爻→展示对应动爻的爻辞（多动爻时全部列出）。此规则在 spec 固化、可测。

## 8. 六十四卦数据集

公版内容，结构化为带 TS 类型的数据。每卦：

```ts
interface HexagramData {
  order: number          // 卦序（King Wen，1–64）
  name: string           // 卦名，如「地天泰」
  symbol: string         // 卦符 unicode，如 ䷊
  upper: string          // 上卦（八经卦）
  lower: string          // 下卦
  lines: boolean[]       // 六爻阴阳，自下而上（true=阳）
  judgment: string       // 卦辞
  lineTexts: string[]    // 六条爻辞，自下而上，长度必须为 6
  shiYao: number         // 世爻位（1–6）
  yingYao: number        // 应爻位（1–6）
}
```

**数据完整性校验（构建期 + 测试）**：恰好 64 卦、卦序 1–64 无重复、每卦 `lines` 长度 6、`lineTexts` 长度 6、世应位 ∈ [1,6]、二进制键唯一覆盖全部 64 种组合。

## 9. 组件设计要点

- **Hexagram.tsx**：纯展示组件，输入「六爻 + 动爻标记 + 世应位」，输出墨金风格的卦象；摇卦页、出卦页、分享卡三处复用。
- **CastingStage.tsx**：用 Framer Motion 编排——逐次抛掷三铜钱、落定、对应爻自下而上点亮成形；可跳过/快进。
- **ShareCard.tsx**：离屏（绝对定位移出视口或放在隐藏容器）渲染，固定尺寸（适配竖屏分享，如 1080×1920 的等比），含：所问之事、卦名、卦象、日期、站点水印 / 印章。

---

## 10. 分享卡片（DOM → PNG）

- 用 `modern-screenshot` 对 ShareCard 栅格化，`pixelRatio` 2–3 保证清晰。
- **截图前确保字体就绪**：`await document.fonts.ready`，避免衬线字未加载导致回退字体。
- 输出后：优先 `navigator.share`（带 file，移动端原生分享）；不支持则提供下载链接，并提示「长按图片保存」。

---

## 11. 错误处理

- **空的所问之事**：**必填**。未填写时禁用「摇卦」按钮，并给出温和的内联提示（如「心诚则灵，先写下所问」）；所问之事随后必定出现在分享卡上。
- **分享图生成失败**（字体未就绪 / 浏览器兼容）：兜底 toast「生成失败，可长按图片保存」，不静默吞错。
- **数据集异常**：构建期与测试期校验拦截，运行时查卦保证命中（64 卦覆盖全部组合）。
- 所有系统边界（用户输入、外部数据）先校验再处理。

## 12. 部署

- Vite `base: '/<仓库名>/'`（GitHub Project Pages 子路径）。
- GitHub Actions：push 到主分支 → 安装依赖 → build → 上传 Pages artifact → 部署。
- 全部静态资源（含子集化字体）本地打包，无外部运行时依赖。

- **单元**：
  - 起卦：三铜钱→6/7/8/9 映射、阴阳/动静判定、概率分布（注入 mock 随机源做确定性断言）。

  - 查卦：二进制→卦正确；世应位正确。

  - 数据集完整性（§8 校验项）。
- **集成**：注入 mock 随机源，跑通 `输入→摇卦→出卦`；ShareCard 渲染内容正确（含所问之事）。
- **E2E（Playwright）**：完整 happy path——输入问题 → 摇卦 → 出卦 → 生成分享图。通过随机源接缝保证确定性。
